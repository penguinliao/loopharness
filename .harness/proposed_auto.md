# Hermes 自反思提议（待审核）

- **失败模式收敛** — 三条"自动回炉/带病交付/永不挂起"必须互斥且穷尽：任一时刻 pipeline 只能落到这三态之一，禁止出现"既不回炉也不交付也不挂起"的卡死缝隙，需要有兜底 default 分支。
- **预算口径单一可信** — "预算耗尽带病交付"依赖一个权威的预算/轮次计数来源；fresh worker 每轮重启不能丢失累计计数（落文件，不靠会话记忆），否则计数器归零→永远到不了"耗尽"→违反 AC3 的永不挂起。
- **外测进程真隔离** — AC5 "干净上下文"必须是物理隔离（独立 agent / 子进程只喂 spec+产物），不能是同一会话里"假装忘记"；要有可验证标志证明 external agent 没读到实现细节或自报内容。
- **fail-open 不等于静默放行** — 外测 fail-open 时必须显式落盘"未验证/降级"原因并让 PM 可见，禁止异常被吞掉后当作 PASS（否则 AC2"永不改判"被一个空 external_review 绕过）。
- **错题本注入可证伪** — AC4"每轮必读错题本"需要机械可验证（worker 派工指令里物理含错题本内容或路径校验），不能只靠 prompt 文字承诺；空错题本与"未注入"要能区分。
- **浊龙旁证不得渗入门禁** — AC1/AC2 要求放行只看 external_review，需防止浊龙报告通过"被 external agent 读到"间接影响判定——external agent 的输入清单必须显式排除 zhuolong_report。
- **派工指令有界** — AC4"有界派工指令"要明确 token/文件数上限与裁剪规则，避免 fresh-context 退化成又把全部历史塞回去（重新产生 2.4× re-read 税，背离本任务初衷）。
