# 外测：独立验收（独立 agent，不押注浊龙）

> 内测 = Claude 自己写测试 + 自跑 + 自检（自报，便宜，拦显眼问题）。
> 外测 = **独立 agent 验收**：不信任自报，从零对照 spec 的验收标准核一遍。
> 独立性本身就是全部价值——它专抓"自以为做完了、其实没做完"。

## 怎么启用

SPEC 阶段写了 `.harness/external_brief.md` → 本任务需要外测。
TEST 阶段内测全过后，由主 Agent **派一个独立验收子 agent**产出 `.harness/external_review.md`，
pipeline 的放行/回炉**只看这份独立验收**。

## 给独立验收 agent 的硬约束

1. **认知隔离 —— 只看 spec + 产物，看不到实现过程。**
   只喂它 `.harness/spec.md`（验收标准）+ 最终产物（代码 diff / 跑起来的应用 / 接口返回）。
   **不要**告诉它代码是怎么写出来的、不要给它实现思路、不要给它内测怎么过的。
   它在干净上下文里独立判断，避免"运动员当裁判"的确认偏误。
2. **逐条核 AC，给后果不给空话。** 对 spec 每条 P0 AC 给"满足 / 不满足 + 证据"，
   不满足要写清"用户会看到什么后果"，不能只写"可信度：高"。
3. **判定写入 `.harness/external_review.md`，末段给明确 PASS / FAIL。**
   有任何一条 P0 AC 不满足 → FAIL（会自动回炉，原因进错题本）。

## 浊龙黑盒：仅旁证，永不改判

浊龙 UI 自动化最近一次实测埋雷发现率仅 ~20%，可靠性不足以当放行依据。
- 仅 Web 应用可跑浊龙，截图作为"用户真能看到"的**旁证**写进报告。
- **浊龙说 PASS 不能放水、说 FAIL 不能拦**——放行与否只由上面的独立验收 verdict 决定。

## fail-open：验收工具不可用时不阻塞

独立验收 agent / 其依赖的工具（CLI、浏览器、网络）不可用时，
**记日志、不阻塞主流程**（与二审/三审同策略：宁可放行也不卡死 PM）。
"验收没法跑"是工具问题，不该让 PM 的交付永久挂起——但要在交付报告里诚实标注"外测未覆盖"。
